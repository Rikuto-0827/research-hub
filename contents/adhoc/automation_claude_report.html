2. Claude 活用の最適化

モデルの使い分け（コスト対効果）

用途推奨モデル理由
ファイル統合・整理・タブUI構築などの「配管」Opus 4.8機械的作業に十分。最上位モデルは過剰でコスト増
PE分析・業種選定・企業分析などの「中身」生成Fable 5（ここぞ）金融分析・長時間の自律作業に優位。分析品質が直接効く場面に限定投入
定型の要約・分類・軽処理Sonnet / Haiku速度・コスト最適。バッチ処理向き

再現性：スキル化（最重要レバー）

「今後の更新が目的」の成果物は、HTMLを作り直すのではなくスキルを叩き直すのが正解。対象・業種・期間を引数化すれば、同じ品質を差し替え再現できる。PEワークフロー＝実装済米国株業種選定＝候補PE市場動向＝候補モーニングブリーフ＝候補

データ使用量の最適化

既存成果物の移行は「再調査せず、確定済み内容を移植」が最小コスト。PDF等の一次成果物がある場合は、その本文を流用してHTML化する（今回の米国株分析タブがこの好例）。

コネクタ／MCP

S&P Capital IQ 等のコネクタは認証が必要（現状は未接続の場面あり）。接続できれば定量データ取得が自動化でき、スクリーニング〜モデル作成の一次情報化が進む。無料一次情報（SEC EDGAR 等）との併用が現実解。

3. ボトルネックと改善機会

論点重大度内容と対応
トークンの平文管理高デプロイ用GitHubトークンがスクリプトに平文＆同期フォルダ内。失効＋再発行し、同期フォルダ外のファイルから読む方式へ（新ハブのdeployスクリプトは対応済）。
成果物の散在（解消済）中各セッションのoutputsに孤立していた成果物を、リサーチハブ＋archiveへ一元化。以後はハブをマスターに。
週次・日次の手動運用中スキル化＋スケジュール実行で無人化余地。型は既に確立。
HTML破損の再発防止低単一巨大HTMLへの追記はリスク。ハブ＝殻／中身分割で影響を局所化済。

4. 推奨アクション（ロードマップ）

P1

GitHubトークンの失効・再発行と安全化

旧トークンを失効し、新トークンを同期フォルダ外（%USERPROFILE%）へ。ハブ公開スクリプトは既にこの方式に対応。MarketBrief側も同方式へ移行可。

P1

週次リサーチのスキル化（米国株業種選定／PE市場動向）

マクロ／ミクロ／示唆の型を引数化。対象・期間を変えて同品質を再現。ハブは最新出力を映すだけにする。

P2

スケジュール実行で無人化

日次ブリーフ・週次リサーチを定期タスク化し、生成→ハブ反映→公開まで自動。

P2

モデル使い分けの徹底

配管＝Opus、中身＝ここぞでFable。定型処理はSonnet/Haiku。用途別にコスト最適化。

P3

コネクタ接続と一次情報化

S&P等の認証を通し、定量取得を自動化。無料一次情報（SEC EDGAR）と併用してディフェンシブルなデータ基盤に。

5. ベストプラクティス集

  • 単発で作らず、型として残す：更新前提の成果物はスキル化し、対象を引数で差し替える。
  • マスターは1か所：成果物はリサーチハブ＋archiveに集約し、セッションoutputsに孤立させない。
  • 更新は局所化：巨大単一HTMLでなく「殻＋中身分割」。タブ単位で差分公開して破損・手戻りを防ぐ。
  • 秘密情報は同期フォルダの外：トークン等はリポジトリにもOneDriveにも置かない。
  • 移行はコスト最小で：既存成果物は再調査せず本文を流用してHTML化。
  • 一次情報で確定：数値はSEC等の一次情報で確定し、ソース差は定義差として検証・調整する。
